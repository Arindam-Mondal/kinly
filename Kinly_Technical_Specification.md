## 1. Project Summary

**Kinly** is a mobile-first Progressive Web App for tracking menstrual cycles. Any user — regardless of sex — logs periods through the same calendar interface, sees the same predictions and analytics, and writes the same notes. The one sex-specific behavior is a copy change on the home dashboard: male users see neutral alert-style framing instead of first-person "your cycle" language, since the data they log is typically about a partner, not themselves.

The system is built so a second logging domain (migraines) and a family/linked-accounts feature can be added later without restructuring the core app — this constraint shapes several decisions below and must not be designed around casually.

## 4. Data Model (summary — full detail in `Kinly_Database_Schema.md`)

Build these two tables first; everything else is additive later.

```sql
create table profiles (
  id uuid primary key references auth.users(id) on delete cascade,
  name text not null,
  email text not null unique,
  age integer not null check (age >= 13 and age <= 120),
  sex text not null check (sex in ('female','male','other','prefer_not_to_say')),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create table log_entries (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references profiles(id) on delete cascade,
  domain text not null check (domain in ('period','migraine','note')),
  start_date date not null,
  end_date date check (end_date is null or end_date >= start_date),
  notes text check (char_length(notes) <= 1000),
  metadata jsonb default '{}'::jsonb,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create index idx_log_entries_user_domain_date on log_entries (user_id, domain, start_date);
create index idx_log_entries_user_date on log_entries (user_id, start_date);

alter table profiles enable row level security;
alter table log_entries enable row level security;

create policy "Users can view own profile" on profiles for select using (auth.uid() = id);
create policy "Users can update own profile" on profiles for update using (auth.uid() = id);
create policy "Users can insert own profile" on profiles for insert with check (auth.uid() = id);

create policy "Users can view own log entries" on log_entries for select using (auth.uid() = user_id);
create policy "Users can insert own log entries" on log_entries for insert with check (auth.uid() = user_id);
create policy "Users can update own log entries" on log_entries for update using (auth.uid() = user_id);
create policy "Users can delete own log entries" on log_entries for delete using (auth.uid() = user_id);
```

RLS is mandatory on every table from the first migration — do not defer this "until later."

**On `domain = 'note'` vs. notes attached to a period:** a standalone note is its own `log_entries` row with `domain = 'note'` and `end_date = null`. A note *about* a specific period is just that period's own `notes` column — there is no separate notes table. See `Kinly_Database_Schema.md` Section 7 if this distinction is unclear.

## 5. Screen-by-Screen Build Spec

### 5.1 Auth: Register
**Route:** `/register`
**Fields:** name (text), email (email), password (min 8 chars, 1+ number, strength indicator), age (number, 13–120), sex (select: Female / Male / Other / Prefer not to say).
**Behavior:** on submit, call Supabase Auth sign-up, then insert the corresponding `profiles` row with the same `id`. On success, redirect to `/onboarding`. Validation errors shown inline, not as a toast.

### 5.2 Auth: Login
**Route:** `/login`
**Fields:** email, password. Standard Supabase Auth sign-in. "Forgot password" link goes to `/forgot-password`, which triggers Supabase's built-in reset email flow.

### 5.3 Onboarding
**Route:** `/onboarding`
Three skippable screens: (1) what Kinly does, (2) one line explaining logging works the same whether it's your own data or someone else's, (3) optional "log your last period now" CTA → opens the Calendar's save-period bottom sheet, or "skip" → goes to `/home`.

### 5.4 Home Dashboard
**Route:** `/home`
Composed of:
- **Header card** — text comes from `getDashboardCopy(sex, predictionData)` (see Section 7 below). Do not hardcode this branch inline in the page component.
- **Mini calendar preview** — current month, tapping navigates to `/calendar`.
- **Quick actions** — "Log Period" (opens calendar's save sheet), "Add Note."
- **Insight teaser card** — one rotating fact or an active delay/skip flag.

### 5.5 Calendar
**Route:** `/calendar`
This is the most interaction-heavy screen — build carefully:
- Month grid, swipeable between months.
- **Tap interaction state machine:**
  1. No selection → tap a date → that date becomes `pendingStart`, highlighted.
  2. `pendingStart` set → tap a later date → becomes `pendingEnd`; all days between get the "selected range" visual treatment (rounded caps at start/end).
  3. `pendingStart` set → tap an *earlier* date than `pendingStart` → that earlier date replaces `pendingStart` (don't force the user to cancel and restart).
  4. Once both dates are set, show a **bottom sheet**: Start Date, End Date (both editable), Notes (optional, max 1000 chars), Save button. Saving inserts a `log_entries` row with `domain = 'period'`. Do not auto-save on the second tap — always confirm via the sheet.
- **Day detail** (long-press or info icon per day): opens a bottom sheet showing that day's note (if any) and lets the user add/edit a note for that specific day, independent of period logging.
- **Visual states to support:** logged period (solid fill), predicted period (dashed/lighter outline), today (ring/border), selected-but-unsaved range (distinct from both of the above).
- **Legend**: small persistent or tap-to-reveal legend explaining the visual states above.
- Past months' logged periods are editable/deletable via the same day-detail/bottom-sheet pattern.

### 5.6 Insights
**Route:** `/insights`
- Cycle length trend chart (Recharts line/bar, last 6–12 cycles, average line overlaid).
- Period duration trend chart (same treatment).
- Compact heatmap/history view of the last 6–12 months.
- Stat cards: average cycle length, average period length, shortest/longest cycle, total cycles logged.
- Written irregularity summary (plain sentence, generated from the same prediction logic as the dashboard flag — see Section 7).
- All copy here is domain-descriptive ("cycle length," "period duration") rather than "your cycle" — this is what lets the screen work identically regardless of who's logging, without a second variant.

### 5.7 Notes / Journal
**Route:** `/notes`
Chronological list (newest first) of all `log_entries` notes across domains, using the shared `NotesList` component. Each item shows date, a domain badge if more than one domain exists (irrelevant until migraines ship, but build the badge slot now so it's a non-event later), and the note text. Tapping opens the same day-detail bottom sheet used from the Calendar.

### 5.8 Settings
**Route:** `/settings`
Edit name/age/sex, dark mode toggle, "Export my data" (downloads JSON via `/api/export`), "Delete account" (confirmation modal, cascades via the DB's `ON DELETE CASCADE` — no manual cleanup needed in application code).

## 6. Prediction Engine Spec

This is business logic, not UI — implement as a pure, testable module at `lib/domains/period/periodInsights.ts`, callable from both a Server Component (for initial render) and a Route Handler (for client-side refetch after a save).

### 6.1 Inputs
All `log_entries` rows for a given `user_id` where `domain = 'period'`, ordered by `start_date` ascending.

### 6.2 Core calculations
```
cycleLength(periodA, periodB) = periodB.start_date - periodA.start_date   // in days
avgCycleLength = average of cycleLength() over the last 6 consecutive period pairs
                 (fewer than 6 if less history exists; need at least 2 periods total to compute anything)
avgPeriodDuration = average of (end_date - start_date + 1) over the last 6 periods
predictedNextStart = mostRecentPeriod.start_date + avgCycleLength
predictedRangeEnd = predictedNextStart + avgPeriodDuration - 1
```

### 6.3 Status / flag logic
Compute `daysSincePredicted = today - predictedNextStart` (only meaningful if no period has been logged since `predictedNextStart`):

| Condition | Status | Dashboard copy tone |
|---|---|---|
| Fewer than 2 periods logged | `insufficient_data` | "Log a couple more cycles to start seeing predictions." |
| `daysSincePredicted <= 4` (or in the future) | `on_track` | Normal — show predicted date |
| `5 <= daysSincePredicted <= 13` | `running_late` | Soft, calm: "This is X days later than the average cycle." |
| `daysSincePredicted >= 14` | `significant_delay` | Calm but slightly more direct; soft suggestion to consult a provider if this continues |
| `today >= predictedNextStart + avgCycleLength` (a full extra cycle has elapsed with nothing logged) | `possible_skip` | "It looks like a cycle may have been skipped this month." |

Flags are **dismissible per-instance** (store a dismissed-until timestamp or similar; exact mechanism is an implementation detail) but should reappear if the same pattern persists into a new analysis window.

**Tone constraint (non-negotiable):** never use alarming language, all-caps, or anything implying a diagnosis. Always frame as informational, always end significant-delay copy with a soft, non-pushy suggestion to talk to a healthcare provider if the pattern continues — never an imperative ("see a doctor now").

### 6.4 Where this runs
Recompute on every period save/edit/delete, synchronously, in the same Server Action / Route Handler that wrote the change (this is cheap — at most 6 rows of arithmetic). Do not defer this to a background job.

**[DECISION NEEDED, carried over from the schema doc]:** whether to persist the result in a `cycle_predictions` cache table or compute it fresh on every read. Both are fine; if deferred, just call `periodInsights.ts` directly from the Home/Insights Server Components on each load. If you want the cache table, its schema is in `Kinly_Database_Schema.md` Section 3.

## 8. Extensibility Contract (read before touching `Calendar.tsx` or `NotesList.tsx`)

These two components must remain domain-agnostic. Concretely:

- `Calendar` accepts entries and a config object (colors/labels for "logged," "predicted," etc.) as props — it has no knowledge of what a "period" is.
- `NotesList` accepts a `domain` filter prop (or `undefined` for all domains) — it has no period-specific logic.
- Domain-specific business logic (predictions, flagging, chart generation) lives in `lib/domains/<domain>/`, one folder per domain, never inside the shared components above.
- When the migraine domain is eventually built, the expected diff is: a new `lib/domains/migraine/` folder, a new `metadata` shape used only by `domain = 'migraine'` rows, new dashboard/insights views that reuse `Calendar` and `NotesList`, and a new `domain` value added to the `log_entries` check constraint. **No changes to `profiles`, the core `log_entries` columns, or the shared components.** If implementing a future domain requires touching any of those, stop and reconsider the approach before proceeding.

## 9. PWA Requirements

- `app/manifest.ts` (Next.js App Router manifest route) — name, short_name, icons (192/512 + maskable variant), `display: "standalone"`, theme_color, background_color.
- Service worker (via `next-pwa` or manual): cache the app shell for offline load; queue writes made while offline and flush on reconnect (exact queueing mechanism is an implementation choice — a simple IndexedDB outbox is sufficient).
- "Add to Home Screen" prompt triggered after the user has logged at least one period — not on first load.
- Lighthouse PWA audit should pass (installability, viewport meta tag, fast load).
- Primary breakpoint: 360–430px width. Don't let layout break on tablet/desktop — a centered max-width container is fine for larger viewports.

## 10. Explicit Non-Goals for This Build

Do not implement these now, even if they seem like small additions while building something adjacent:
- No caretaker/observer mode or any second dashboard variant beyond the Section 7 copy branch.
- No `profile_links` table, no linked-account UI.
- No push notifications.
- No AI report analysis (`report_analyses` table, `/api/reports/analyze`) — the route file can exist as an empty stub per Section 3's folder structure, but do not implement the logic.
- No migraine domain implementation — only the folder convention (Section 8) needs to exist, not the feature.
- No multi-device sync beyond what Supabase gives for free (it gives this for free, so this isn't actually extra work — just don't build anything bespoke on top of it).

## 11. Build Sequence

1. Scaffold Next.js (App Router, TypeScript) + Tailwind + PWA manifest/service worker.
2. Supabase project setup: Auth enabled, `profiles` + `log_entries` tables, RLS policies (Section 4).
3. Auth flows: register, login, forgot-password — wired to Supabase Auth, writing the `profiles` row on signup.
4. `Calendar` component: month grid, tap-to-select state machine, bottom-sheet confirm (Section 5.5) — built domain-agnostic from the start (Section 8).
5. Period logging on top of `Calendar`: save/edit/delete `log_entries` rows with `domain = 'period'`.
6. Prediction engine (`periodInsights.ts`, Section 6) as a standalone module.
7. Home Dashboard, including `getDashboardCopy` (Section 7).
8. `NotesList` component + Notes/Journal screen + day-detail bottom sheet.
9. Insights screen with Recharts visualizations.
10. Settings: dark mode, export (`/api/export`), delete account.
11. Onboarding flow.
12. Polish: empty states, install prompt, offline write queueing, Lighthouse pass.
13. Sanity check against Section 8's extensibility contract — confirm a hypothetical migraine domain could be added by only touching new files, not the components built in steps 4 and 8.
