Key principle: the synchronous request path (solid grey arrows) never includes the AI call. A report upload returns immediately; the actual analysis happens in a background Edge Function and the result streams back via Supabase Realtime. 02 Request Flow — Logging a Period (Synchronous, MVP)

This entire flow runs inside a single Server Action / Route Handler call — no queue, no async worker. The prediction engine (Section 7.6 of the PRD) runs as plain server-side logic immediately after the insert, since it's just arithmetic over recent rows, not an AI call.

03 Request Flow — AI Report Analysis (Phase 2, Asynchronous)

Why this avoids "backend-heavy" problems: the Next.js Route Handler in step 2–3 does almost no work — it stores the file and returns. All the slow, potentially-expensive work (steps 4–7) happens in an Edge Function that isn't tied to a user-facing HTTP request's timeout. This pattern scales the same way whether you get 10 report uploads a day or 10,000.

04 Why This Doesn't Become "Backend-Heavy"

What stays in Next.js

  • All UI rendering (Server + Client Components)
  • Auth session handling (via Supabase SSR helpers)
  • Fast CRUD: save period, add note, edit profile
  • Prediction engine (pure arithmetic, runs in milliseconds)
  • Kicking off a report-analysis job (fire-and-forget, <1s response)
  • Reading job status / subscribing to the result

What moves to async workers

  • Calling the Claude API to analyze an uploaded report
  • Any future long-running batch job (e.g., monthly summary emails)
  • Heavy file processing (OCR, PDF parsing) ahead of the AI call
  • Anything that could exceed a normal HTTP request's time budget
The rule of thumb: if a task finishes in well under a second, it can live directly in a Next.js Route Handler or Server Action. If it might take seconds to minutes (an AI call almost always does), it gets handed off to an Edge Function / worker and the user-facing request returns immediately. This single pattern is what lets the same Next.js app handle both the simple CRUD of period logging and a heavier AI feature later — without needing a separate Express backend.

Row Level Security is enabled on every table. Base policy: a user reads/writes only rows where user_id = auth.uid(). profile_links adds a second policy layer later — granting read access to a linked profile's rows — without altering the base policy on log_entries.

06 Scaling Notes

App layer (Next.js)

  • Stateless — scales horizontally by running more instances
  • Static assets + cacheable pages served via CDN
  • Self-hostable (Docker / any cloud) if outgrowing a serverless platform's limits

Data layer (Postgres)

  • Index log_entries(user_id, domain, start_date) early
  • Connection pooling (built into Supabase) handles high concurrent connections
  • Read replicas available on higher tiers if read load grows
  • Active concurrent load is typically far lower than total registered users for a low-frequency-logging app like this
